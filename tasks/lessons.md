# Lessons

- Trigger: Performing a non-trivial repository code review with ratings. Action: use parallel subagents for mapping, risk review, and targeted test execution, then validate top findings directly in source before reporting severity. Verification: include at least one passing focused test command plus line-referenced evidence for each high-severity finding.
- Trigger: Fixing async workflow state bugs (cancel/race/claim ownership). Action: enforce ownership/state with conditional DB transitions (`pending -> running` only, claim-token match required for claimed outbox rows) and preserve terminal states as immutable. Verification: add tests for stale claim-token rejection, cancelled execution no-op behavior, and successful processing only with current claim token.
- Trigger: Reviewing or hardening persisted workflow runtimes. Action: enforce engine state-contract guards (`resume` only from `waiting`, `cancel` only from active states) and keep retry budgets tied to actual delivery failures rather than claim/lease churn. Verification: add regression tests for resume/cancel terminal-state rejection and for repeated stale-claim reclamation that does not exhaust retries without handler execution.
- Trigger: Hardening workflow correlation or registration dedupe semantics. Action: pair application-level dedupe with a DB uniqueness constraint for the durable key, and explicitly deduplicate identical registrations at registration time so local and production paths match. Verification: add tests for in-flight correlation-id reuse, failed-execution reuse policy, and duplicate identical handler registration executing exactly once in both registries.
- Trigger: Assessing production-load readiness for asynchronous workflow systems. Action: combine static hot-path analysis with explicit checks for queue fan-out, lock contention, scheduler/retry wake-up behavior, and concurrency idempotency under at-least-once delivery. Verification: run focused workflow suite and explicitly note missing perf/concurrency stress coverage before giving readiness verdicts.
- Trigger: Hardening unreleased async workflow runtimes for production. Action: enforce strict importable-handler requirements in async mode (no inline fallback), wire periodic Beat outbox draining as core behavior, and validate lock-protected claim ownership paths with duplicate-suppression tests. Verification: run targeted workflow unit/integration tests plus mypy on touched workflow modules and ensure deterministic pass before handoff.
- Trigger: Testing fallback paths that depend on an attribute being absent (for example Celery task `.delay`). Action: use a real callable or a constrained mock instead of a plain `Mock()`, because `Mock` will auto-create missing attributes and can accidentally exercise the wrong branch. Verification: assert the fallback callable itself was invoked and rerun the targeted regression test.
- Trigger: Reserving correlation or idempotency keys before inline work completes in an in-memory runtime. Action: pair the reservation with an explicit completion signal or wait path so concurrent callers do not observe placeholders as final state. Verification: add a two-thread regression test where the first execution blocks mid-handler and the second correlated start returns the finalized execution once the first completes.
- Trigger: Adding workflow-state constraints in models or migrations. Action: derive the allowed state set from shared engine constants instead of hardcoding string tuples so DB constraints stay aligned with runtime semantics. Verification: update the shared constant and run focused backend tests plus touched-file checks after wiring the constraint to it.
- Trigger: Verifying custom `AUTH_USER_MODEL` behavior that depends on startup auto-imports. Action: prefer a dedicated subprocess or settings module over in-process `override_settings(INSTALLED_APPS=..., AUTH_USER_MODEL=...)` so app-registry state and auth-model resolution cannot leak across tests. Verification: run the focused integration scenario in the isolated process and rerun adjacent existing-model tests in-process.
- Trigger: Remediating dependency security alerts in repository manifests. Action: update both pinned requirement files and `pyproject.toml` dependency floors so checked-in installs and published metadata stop allowing vulnerable versions. Verification: run a focused Django/import-sensitive pytest slice after the version bump and record any non-applicable lint checks explicitly.
