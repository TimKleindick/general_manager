# Lessons

- Trigger: Performing a non-trivial repository code review with ratings. Action: use parallel subagents for mapping, risk review, and targeted test execution, then validate top findings directly in source before reporting severity. Verification: include at least one passing focused test command plus line-referenced evidence for each high-severity finding.
- Trigger: Fixing async workflow state bugs (cancel/race/claim ownership). Action: enforce ownership/state with conditional DB transitions (`pending -> running` only, claim-token match required for claimed outbox rows) and preserve terminal states as immutable. Verification: add tests for stale claim-token rejection, cancelled execution no-op behavior, and successful processing only with current claim token.
- Trigger: Reviewing or hardening persisted workflow runtimes. Action: enforce engine state-contract guards (`resume` only from `waiting`, `cancel` only from active states) and keep retry budgets tied to actual delivery failures rather than claim/lease churn. Verification: add regression tests for resume/cancel terminal-state rejection and for repeated stale-claim reclamation that does not exhaust retries without handler execution.
- Trigger: Hardening workflow correlation or registration dedupe semantics. Action: pair application-level dedupe with a DB uniqueness constraint for the durable key, and explicitly deduplicate identical registrations at registration time so local and production paths match. Verification: add tests for in-flight correlation-id reuse, failed-execution reuse policy, and duplicate identical handler registration executing exactly once in both registries.
- Trigger: Assessing production-load readiness for asynchronous workflow systems. Action: combine static hot-path analysis with explicit checks for queue fan-out, lock contention, scheduler/retry wake-up behavior, and concurrency idempotency under at-least-once delivery. Verification: run focused workflow suite and explicitly note missing perf/concurrency stress coverage before giving readiness verdicts.
- Trigger: Hardening unreleased async workflow runtimes for production. Action: enforce strict importable-handler requirements in async mode (no inline fallback), wire periodic Beat outbox draining as core behavior, and validate lock-protected claim ownership paths with duplicate-suppression tests. Verification: run targeted workflow unit/integration tests plus mypy on touched workflow modules and ensure deterministic pass before handoff.
- Trigger: Testing fallback paths that depend on an attribute being absent (for example Celery task `.delay`). Action: use a real callable or a constrained mock instead of a plain `Mock()`, because `Mock` will auto-create missing attributes and can accidentally exercise the wrong branch. Verification: assert the fallback callable itself was invoked and rerun the targeted regression test.
- Trigger: Reserving correlation or idempotency keys before inline work completes in an in-memory runtime. Action: pair the reservation with an explicit completion signal or wait path so concurrent callers do not observe placeholders as final state. Verification: add a two-thread regression test where the first execution blocks mid-handler and the second correlated start returns the finalized execution once the first completes.
- Trigger: Adding workflow-state constraints in models or migrations. Action: derive the allowed state set from shared engine constants instead of hardcoding string tuples so DB constraints stay aligned with runtime semantics. Verification: update the shared constant and run focused backend tests plus touched-file checks after wiring the constraint to it.
- Trigger: Verifying custom `AUTH_USER_MODEL` behavior that depends on startup auto-imports. Action: prefer a dedicated subprocess or settings module over in-process `override_settings(INSTALLED_APPS=..., AUTH_USER_MODEL=...)` so app-registry state and auth-model resolution cannot leak across tests. Verification: run the focused integration scenario in the isolated process and rerun adjacent existing-model tests in-process.
- Trigger: Remediating dependency security alerts in repository manifests. Action: update both pinned requirement files and `pyproject.toml` dependency floors so checked-in installs and published metadata stop allowing vulnerable versions. Verification: run a focused Django/import-sensitive pytest slice after the version bump and record any non-applicable lint checks explicitly.
- Trigger: Fixing mypy failures caused by optional third-party packages. Action: prefer installing typed dev-only stubs when available (for example `celery-types`) and keep suppressions narrowly scoped to the remaining untyped import sites or vendor namespaces instead of broad repo-wide ignores. Verification: rerun full `mypy` plus the smallest focused pytest slice covering the touched integration boundaries.
- Trigger: Making Django migrations satisfy both mypy and Ruff. Action: keep `Migration.dependencies` and `Migration.operations` as typed `ClassVar[list[...]]` values instead of tuples so Django-stubs and `RUF012` both pass without runtime behavior changes. Verification: rerun full `mypy` and a focused migration-importing pytest slice after the edit.
- Trigger: Changing GeneralManager lifecycle semantics after writes or deletes. Action: guard every public field-read surface (`manager.attr` and `dict(manager)`/iteration), refresh relation-backed accessors in place after updates, and verify the `post_data_change.instance` identity contract explicitly. Verification: run focused manager unit/integration tests that cover scalar refresh, relation refresh, post-update signal identity, and invalidation after delete.
- Trigger: Creating or updating a local Codex skill. Action: check whether the target skill directory already exists, stage the draft in the workspace, then validate both the staged copy and the installed copy after updating the live skill. Verification: run `quick_validate.py` on the staged path and the final `~/.codex/skills/<skill-name>` path.
- Trigger: Wrapping `settings.AUTH_USER_MODEL` with `ExistingModelInterface` while related managers add more than one FK path back to that user model. Action: key reverse-relation descriptors to the concrete FK field, and prefer suppressing the shared audit FK reverse accessor with `related_name="+"` so domain FKs can stay plain unless they need a custom reverse API. Verification: run an isolated swappable-auth subprocess test plus `manage.py makemigrations --check --dry-run` for the affected project.
- Trigger: Expecting GeneralManager classes to appear in `manage.py shell` auto-imports. Action: override Django's `shell` command import list to swap any model path with its `_general_manager_class` path and append remaining `GeneralManagerMeta.all_classes` entries instead of rebinding the app's `models.py` exports. Verification: run `manage.py shell -c` in a subprocess and assert the imported symbols come from the manager module.
- Trigger: Adding local fallback filtering to request-backed query interfaces. Action: keep filter validation declaration-driven even on materialized buckets, enforce `operation_names` before compiling a request plan, and reject local fallback on partial remote pages where counts/pages cannot stay correct. Verification: add focused tests for materialized-bucket filter validation, operation-restricted filters, and paginated local-fallback failure.
- Trigger: Testing interface-validation failures that occur during capability binding. Action: wrap the dynamic `GeneralManager`/`Interface` class definition itself in the failing assertion instead of calling `get_capabilities()` after class creation, because request-interface validation runs during manager class construction. Verification: rerun the targeted validation test module and confirm the expected exception is raised at definition time.
- Trigger: Logging request-interface result metadata from immutable dataclass wrappers. Action: treat observability metadata as a generic `Mapping`, not only a concrete `dict`, so `MappingProxyType`-backed `RequestQueryResult.metadata` still contributes `status_code`, `retry_count`, and `request_id` to success logs. Verification: rerun the request transport observability integration test and confirm both query and lazy-detail logs include the sanitized request metadata.
- Trigger: Redesigning request-interface declarations that validation reads during class construction. Action: normalize class-attribute `RequestField`s and `Interface.Meta` request config in `RequestInterface.__init_subclass__` so capability binding sees the final request shape before manager lifecycle hooks run. Verification: rerun the request-interface unit and hardening suites and confirm Meta-based interfaces pass while legacy top-level declarations fail at class definition time.
- Trigger: Adding a first-party stdlib HTTP transport. Action: validate the URL scheme before constructing the stdlib request object so the transport rejects `file:` and custom schemes explicitly and passes security lint without hiding the check in a broad suppression. Verification: rerun the transport primitive tests plus `ruff check` on the transport module and confirm only `http`/`https` URLs are accepted.
- Trigger: Cloning a normalized interface subclass during lifecycle pre-create. Action: copy already-normalized interface attributes from the original interface class into the generated clone instead of recomputing only from `Meta`, so synthesized operations/config survive subclass-specific `__init_subclass__` logic. Verification: rerun the focused integration path for the derived interface plus adjacent validation/request suites and confirm the cloned interface resolves the same operations and transport config as the source class.
- Trigger: Testing websocket invalidation against the ASGI remote-manager route. Action: establish the websocket subscription before emitting the mutation/change event, because the invalidation channel does not buffer past events for late subscribers. Verification: rerun the focused remote invalidation integration test and confirm the event is received only when `RemoteInvalidationClient.connect()` happens before the change.
- Trigger: Pickling request-backed buckets that already materialized remote payloads. Action: restore the bucket from serialized raw items instead of relying on pickled manager instances, so field values survive round-trip without reissuing transport requests. Verification: pickle and unpickle a materialized request bucket and assert item values and transport call counts stay unchanged.
- Trigger: Clearing ASGI websocket routes in tests or runtime. Action: rebuild the live `ProtocolTypeRouter` / `AuthMiddlewareStack` from the updated `websocket_urlpatterns` list after filtering remote routes, rather than assuming list mutation updates the active router. Verification: seed a marked remote route alongside an unrelated route and assert the router route count shrinks after `clear_remote_invalidation_routes()`.
- Trigger: Appending mutation capabilities to cloned request interfaces. Action: preserve the existing capability bundle and refresh configured capabilities after copying Meta-derived operations, instead of overwriting the bundle with the clone's default declarations. Verification: rerun remote-manager create/update/delete integration coverage on the cloned request interface and confirm mutation methods remain present.
- Trigger: Recovering patch coverage after centralizing error mapping or async cleanup helpers. Action: add public-entry-point tests for each exception/cleanup branch at the view or client layer instead of testing the helper in isolation, so the new orchestration lines and their user-visible responses are both covered. Verification: rerun the smallest affected pytest slice and confirm it exercises each intended status/cleanup path.
- Trigger: Letting request interfaces accept transports that return either normalized query results or raw transport responses. Action: normalize non-`RequestQueryResult` returns at `RequestInterface.execute_request_plan()` before applying any response serializer, so custom transports and shared transports follow the same serializer/schema path. Verification: add a focused test where `transport.execute()` returns `RequestTransportResponse` and confirm the interface response serializer still runs on the normalized items.
- Trigger: Patching `general_manager.interface.base_interface.InterfaceBase` in unit tests while GraphQL helpers may import `OrmInterfaceBase`. Action: import `general_manager.interface.orm_interface` before the patch takes effect so the real generic base class is defined before `InterfaceBase` becomes a mock. Verification: rerun the affected GraphQL unit module and confirm schema-generation tests no longer fail with an `OrmInterfaceBase` metaclass conflict.
- Trigger: Splitting a legacy permission class into explicit additive and override variants. Action: preserve the legacy additive runtime contract, keep `__based_on__` as a separate outer gate, and make `describe_permissions()` report the effective local rule source for override semantics instead of concatenating ignored base rules. Verification: run focused permission/bootstrap/public-api pytest coverage plus targeted `ruff check` and `mypy` on the touched permission modules.
- Trigger: Hardening GraphQL read authorization that mixes queryset/search prefilters with manager permissions. Action: treat read filters as prefilters only, require a final per-instance row gate for unfilterable rules, `__based_on__` class-context planning, and conflicting delegated/local filter keys, and keep counts/membership derived only from rows that pass that final gate. Verification: run focused manager-permission, GraphQL list/search integration, `ruff check`, and `mypy` coverage on the touched authorization modules.
- Trigger: Adding observability to fail-closed GraphQL permission gates. Action: emit one aggregate structured log at the centralized final row gate, propagate stable `instance_check_reasons` from read-plan construction, and avoid extending the GraphQL metrics backend protocol unless the auth summary fits its existing request/resolver contract. Verification: run focused GraphQL helper/query/search tests plus targeted `ruff check` and `mypy` on the touched permission and GraphQL modules.
- After refactoring shared ORM audit attribution in `OrmMutationCapability` -> run one compatibility slice for existing/custom models in addition to the GeneralManager-backed tests to catch semantic changes for models owning `changed_by` -> rerun the targeted GeneralManager mutation suite and `tests/integration/test_existing_model_manager.py` to verify.
- Trigger: Running focused tests from a Codex worktree when `general_manager` is installed editable from a different checkout. Action: force the worktree `src` onto `PYTHONPATH` for verification commands so pytest imports the modified files instead of the external editable install. Verification: print `general_manager.measurement.measurement.__file__` before the run and confirm the targeted pytest slice passes against the worktree path.
- Trigger: Extending ORM field metadata translation for database-backed interfaces. Action: make Django field-to-Python type translation subclass-aware first, then add explicit mappings only for distinct scalar families that GraphQL already accepts without API changes. Verification: add a focused `build_field_descriptors()` regression that covers inherited integer/string field variants plus explicit `TimeField`, `DurationField`, `UUIDField`, and `BinaryField` mappings.
- Trigger: Preserving GraphQL-safe semantics for ORM big-integer fields. Action: keep the interface `type` as Python `int`, but attach a separate `graphql_scalar` metadata hint and route GraphQL read/write/filter generation through that hint instead of inferring only from the collapsed Python type. Verification: add focused tests for descriptor metadata, `BigIntScalar` round-trip behavior, GraphQL read-field mapping, write-field generation, and generated filter inputs.
- Trigger: Adding GraphQL special-scalar hints for ORM integer fields. Action: exclude `AutoField` subclasses from big-integer scalar hints so auto-generated primary keys preserve the existing GraphQL `Int` contract while non-auto `BigIntegerField` families can still use `BigIntScalar`. Verification: add a focused descriptor test for `BigAutoField` IDs and rerun the affected default-mutation integration tests plus the full pytest suite.
- Trigger: Implementing integer-like GraphQL custom scalars in Python. Action: reject boolean inputs explicitly in scalar `serialize` and `parse_value` methods before coercing with `int(...)`, because `bool` is a subclass of `int` and otherwise slips through as `1`/`0`. Verification: add focused scalar tests for `True`/`False` rejection plus literal parsing coverage for both `StringValueNode` and `IntValueNode`.
